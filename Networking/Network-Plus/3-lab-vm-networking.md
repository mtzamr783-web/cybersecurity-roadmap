<h2 dir="rtl" align="right" id="download-vmware">1️⃣ تحميل برنامج المحاكاة (VMware)</h2>

<div dir="rtl">

**VMware** هو برنامج Virtualization بيسمحلك تشغل أجهزة كمبيوتر وهمية (Virtual Machines) جوه جهازك الحقيقي.

- بيتحمل من الموقع الرسمي وبيتركب على نظام التشغيل الأساسي بتاعك.
- بعد التركيب بيظهرلك واجهة فيها خيارات إنشاء وإدارة الأجهزة الافتراضية.
- الجهاز الحقيقي بتاعك اللي شغال عليه VMware بيتسمى **Host Machine**، والأجهزة الوهمية اللي بتشغلها جواه بتتسمى **Guest Machines**.

</div>

---

<h2 dir="rtl" align="right" id="typical-vs-custom">2️⃣ إنشاء Virtual Machine — الفرق بين Typical و Custom</h2>


لما بتنشئ VM جديدة في VMware، بيدّيك خيارين:

<h3 dir="rtl" align="right">✅ Typical</h3>

- طريقة سريعة وبسيطة للمبتدئين.
- VMware بيختار الإعدادات الافتراضية تلقائيًا (حجم الرام، الهاردديسك، كارت الشبكة...).
- مناسبة لو مش محتاج تتحكم في التفاصيل.

<h3 dir="rtl" align="right">⚙️ Custom (Customize)</h3>

- بتختار كل إعداد بنفسك بشكل يدوي خطوة بخطوة:
  - إصدار نظام التشغيل.
  - حجم الذاكرة (RAM).
  - حجم الهاردديسك الافتراضي.
  - نوع كارت الشبكة (Network Adapter).
- مفيدة لو عايز تتحكم في تفاصيل الجهاز بالكامل.

<h2 dir="rtl" align="right" id="create-two-vms">3️⃣ إنشاء 2 Virtual Machines</h2>


- تم إنشاء **جهازين افتراضيين** على نفس الـ Host.
- كل جهاز عنده نظام تشغيل خاص بيه وإعدادات مستقلة.
- الجهازين شغالين في نفس الوقت على نفس الجهاز الحقيقي.

<h2 dir="rtl" align="right" id="connect-vms-vswitch">4️⃣ توصيل الجهازين ببعض (Physical Connection via Virtual Switch)</h2>


علشان جهازين يتكلموا مع بعض، لازم يكونوا على نفس الشبكة.

- في VMware فيه **Virtual Switch** (سويتش وهمي) اسمه عادةً **VMnet**.
- لما بتوصل الجهازين بنفس الـ VMnet، يبقوا كأنهم متوصلين بنفس السويتش الفيزيائي في الواقع.
- ده اللي بيسمح للجهازين يتبادلوا البيانات مع بعض.

> **مثال**: لو الجهاز الأول والجهاز التاني كلهم على **VMnet1**، يبقوا في نفس الشبكة المحلية ويقدروا يوصلوا لبعض.

<h3 dir="rtl" align="right">أنواع الـ VMnet (وضع الاتصال في VMware)</h3>


مش كل الـ VMnet بتشتغل بنفس الطريقة؛ فيه 3 أوضاع اتصال أساسية لازم تفرق بينهم:

<div dir="rtl">

| الوضع | الاسم | الوظيفة |
|---:|---:|---:|
| **Bridged** | VMnet0 (افتراضيًا) | الجهاز الوهمي بياخد IP حقيقي من نفس شبكة الـ Host الفعلية (زي أي جهاز إضافي على الراوتر)، وكأنه جهاز مستقل تمامًا على الشبكة الحقيقية |
| **NAT** | VMnet8 (افتراضيًا) | الجهاز الوهمي بياخد IP داخلي خاص، وبيطلع للإنترنت عن طريق مشاركة الـ IP بتاع جهاز الـ Host نفسه (زي أي جهاز خلف راوتر منزلي) |
| **Host-only** | VMnet1 (افتراضيًا) | الجهاز الوهمي بيتواصل بس مع جهاز الـ Host ومع باقي الأجهزة الوهمية على نفس الـ VMnet، ومعزول تمامًا عن أي شبكة خارجية أو الإنترنت |

</div>

> **ملحوظة:** اللاب ده استخدم وضع **Host-only** (أو VMnet مخصص معزول) عشان يضمن إن الجهازين معزولين تمامًا عن أي DHCP خارجي، وده اللي خلانا نحتاج نحدد الـ IP يدويًا في الخطوة الجاية.

<h2 dir="rtl" align="right" id="change-ip-manually">5️⃣ تغيير الـ IP Address يدويًا</h2>


<h3 dir="rtl" align="right">ليه بنغير الـ IP يدوي؟</h3>

لأنه مفيش DHCP Server في الشبكة الوهمية دي يوزع العناوين تلقائيًا، فلازم نحددها بنفسنا.

> 💡 **ملاحظة مرتبطة:** لو الجهاز كان متوقع ياخد IP من DHCP ومفيش DHCP Server شغال أو متاح، Windows كان هيدّي الجهاز عنوان **APIPA** تلقائيًا (نطاق `169.254.x.x`) بدل ما يفضل من غير IP خالص، لكن العنوان ده كان هيخلي الجهازين يتواصلوا مع بعض بس من غير أي وصول حقيقي لأي شبكة تانية. عشان كده في اللاب فضّلنا نحدد IP يدوي وواضح بدل ما نسيب الموضوع لـ APIPA.

<h3 dir="rtl" align="right">خطوات تغيير الـ IP في Windows:</h3>

1. **Control Panel** ← **Network and Sharing Center**
2. **Change Adapter Settings**
3. كليك يمين على كارت الشبكة ← **Properties**
4. اختار **Internet Protocol Version 4 (TCP/IPv4)**
5. اختار **Use the following IP address**
6. ادخل الـ IP يدوي.

<h3 dir="rtl" align="right">الـ IPs اللي اتعملت:</h3>

- **الجهاز الأول**: مثلًا `192.168.1.1` ← Subnet Mask: `255.255.255.0`
- **الجهاز التاني**: مثلًا `192.168.1.2` ← Subnet Mask: `255.255.255.0`

> **ملاحظة مهمة**: الجزء الأول من الـ IP (الثلاث أرقام الأولى `192.168.1`) لازم يكون **نفسه** في الجهازين لأنه بيمثل الشبكة. الجزء الأخير بس (`.1` و`.2`) هو اللي بيختلف وبيمثل هوية كل جهاز على حدة.

> 💡 **بالمناسبة**: الـ Subnet Mask `255.255.255.0` ممكن كمان تتكتب بصيغة مختصرة أسرع وهي `/24` (صيغة الـ CIDR)، يعني تقدر تكتب الشبكة دي كلها كـ `192.168.1.0/24` بدل كتابة الماسك كامل بشكل منفصل.

<h2 dir="rtl" align="right" id="ip-address-parts">6️⃣ أجزاء الـ IP Address</h2>


الـ IP Address بيتقسم لجزئين أساسيين:

<div dir="rtl">

| الجزء | المثال | الاسم | الوظيفة |
|---:|---:|---:|---:|
| أول 3 أرقام | `192.168.1` | **Network ID** | بيحدد الشبكة اللي الجهاز فيها |
| الرقم الأخير | `.1` أو `.2` | **Host ID** | بيحدد الجهاز نفسه داخل الشبكة |

</div>

> **القاعدة**: أي جهازين عايزين يتواصلوا لازم يكون الـ Network ID بتاعهم **متطابق**، والـ Host ID **مختلف**.

> 💡 **ملاحظة**: في اللاب ده معملناش Default Gateway، لأننا محتاجين بس الجهازين يتكلموا مع بعض داخل نفس الشبكة المحلية (Same Network ID)؛ الـ Default Gateway كان هيبقى ضروري بس لو أي جهاز محتاج يوصل لشبكة تانية مختلفة تمامًا (زي الإنترنت).

<h2 dir="rtl" align="right" id="disable-firewall">7️⃣ إيقاف الـ Firewall (جدار الحماية)</h2>


<h3 dir="rtl" align="right">ليه اتوقف الـ Firewall؟</h3>

الـ Firewall بيحجب كتير من الاتصالات الواردة بشكل افتراضي كإجراء أمان، ومنها أوامر الـ Ping. علشان نختبر التوصيل بين الجهازين بحرية في اللاب، اتقرر إيقافه مؤقتًا.

<h3 dir="rtl" align="right">خطوات إيقاف الـ Firewall في Windows:</h3>

1. **Control Panel** ← **Windows Defender Firewall**
2. اختار **Turn Windows Defender Firewall on or off**
3. اختار **Turn off** للـ Private Network.

<h3 dir="rtl" align="right">أهمية الـ Firewall (وليه مينفعش توقفه في الواقع):</h3>

- بيحمي جهازك من الاتصالات غير المصرح بيها.
- بيمنع البرامج الضارة من الاتصال بالإنترنت.
- في البيئة الحقيقية **ميوقفش أبدًا** إلا لو عارف بالظبط إيه اللي بتعمله.
- في اللاب بيتوقف بس لأغراض الاختبار والتعلم.

<h2 dir="rtl" align="right" id="test-ping">8️⃣ اختبار التوصيل بأمر Ping</h2>


<h3 dir="rtl" align="right">إيه هو Ping؟</h3>

أمر بيبعت **ICMP packets** صغيرة من جهازك لجهاز تاني ويستنى رد، علشان يتأكد إن الاتصال شغال.

<h3 dir="rtl" align="right">طريقة الاستخدام:</h3>

- افتح **Command Prompt (CMD)** على أي من الجهازين.
- اكتب:
```
ping 192.168.1.2
```
- لو الاتصال شغال هيظهرلك رد زي:
```
Reply from 192.168.1.2: bytes=32 time<1ms TTL=128
```
- لو مفيش اتصال هيظهر:
```
Request timed out.
```

<h3 dir="rtl" align="right">ليه نجح الـ Ping في اللاب؟</h3>

لأن الشروط الثلاثة اتحققت:
1. ✅ الجهازين على نفس الـ VMnet (نفس السويتش).
2. ✅ الـ IP بتاع كل جهاز في نفس الـ Network ID.
3. ✅ الـ Firewall اتوقف فخلى الـ Ping يعدي.

<h2 dir="rtl" align="right" id="diagnostic-commands">9️⃣ أوامر تشخيص إضافية مفيدة (Diagnostic Commands)</h2>


غير أمر الـ `ping`، فيه أوامر تانية أساسية جدًا في أي لاب شبكات، ومفيد تتعود عليهم من دلوقتي:

<h3 dir="rtl" align="right">ipconfig — عرض إعدادات الشبكة الحالية</h3>

بيوريك كل تفاصيل الـ IP الحالية على الجهاز (IP Address، Subnet Mask، Default Gateway) من غير ما تدخل الـ Properties. مفيد جدًا عشان تتأكد بسرعة إن الـ IP اللي ظبطته فعلًا اتسجل صح.

```
ipconfig
```

للتفاصيل الكاملة (زي الـ MAC Address وحالة الـ DHCP)، بتستخدم:

```
ipconfig /all
```

<h3 dir="rtl" align="right">tracert — تتبع مسار الحزمة</h3>

بيوريك كل نقطة (Hop) بتعدي عليها الحزمة من جهازك لحد ما توصل للوجهة، ومفيد جدًا لو الـ Ping بيفشل وعايز تعرف فين بالظبط المشكلة حصلت في المسار.

```
tracert 192.168.1.2
```
